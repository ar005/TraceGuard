# TraceGuard Agent Documentation

## Overview

The TraceGuard Agent (`edr-agent`) is a lightweight, high-performance endpoint telemetry collector for Linux. It runs on monitored endpoints with root privileges and uses eBPF probes to capture process, network, file, and authentication events in real time. Events are published to an internal event bus, buffered locally in SQLite for offline resilience, and streamed to the TraceGuard backend over gRPC.

The agent is written in Go with eBPF C probes compiled via clang/libbpf. It supports hot-reloadable YAML configuration, self-protection (watchdog + binary immutability), and a browser extension receiver for HTTP request visibility.

## Prerequisites

| Requirement | Minimum Version | Notes |
|---|---|---|
| Linux kernel | >= 5.8 | Required for BTF and modern eBPF features |
| Go | 1.22+ | For building the agent binary |
| clang | 14+ | Compiles eBPF C programs to BPF bytecode |
| libbpf-dev | — | eBPF loader library headers |
| Privileges | root or `CAP_BPF` + `CAP_PERFMON` | Required to load eBPF programs and read `/proc` |

## Build Instructions

All build commands are run from the `edr-agent/` directory.

```bash
cd edr-agent

# 1. Verify toolchain (clang, go, libbpf, etc.)
make check-deps

# 2. Compile eBPF C probes (.bpf.c -> .bpf.o)
make ebpf

# 3. Generate Go bindings from compiled BPF objects (bpf2go)
make generate

# 4. Build the agent binary
make build

# Full rebuild from scratch (check-deps -> vmlinux -> ebpf -> generate -> build)
make all

# Run tests
make test       # go test ./... -v -race -count=1

# Lint
make lint       # golangci-lint run ./...
```

**Important:** Do not edit `*_bpfel.go` or `*_bpfeb.go` files -- these are auto-generated by `make generate`.

## Running the Agent

```bash
sudo ./edr-agent --config config/agent.yaml
```

The agent requires root privileges (or `CAP_BPF` + `CAP_PERFMON`) to load eBPF programs and access `/proc` for process enrichment.

On startup, the agent:

1. Loads configuration from YAML (with env var overrides via `EDR_` prefix)
2. Generates or loads a persistent agent ID from `id_file`
3. Auto-detects hostname if not configured
4. Initializes the event bus, local SQLite buffer, and gRPC transport
5. Starts self-protection (watchdog)
6. Starts all enabled monitors
7. Publishes an `AGENT_START` event with build info
8. Sends heartbeats every 30 seconds
9. Starts a live response client for remote command execution

On shutdown (SIGTERM/SIGINT), it publishes an `AGENT_STOP` event, stops monitors in reverse order, flushes the transport, and closes the local buffer.

## Monitors

### Process Monitor

Uses eBPF tracepoints attached to the kernel's process lifecycle to capture execution, exit, fork, and ptrace events. Enriches each event with:

- Full process context (PID, PPID, UID, GID, EUID, command line, exe path, working directory)
- Process ancestry (walks up to `max_ancestry_depth` parents)
- Container context (container ID, runtime, image name, pod name, namespace)
- SHA256 hash of the executed binary
- Open socket enumeration from `/proc/<pid>/fd` at exec time
- Network target extraction from command-line arguments (URLs, hosts)
- Detection of memfd execution (fileless malware)
- Environment variable capture for processes spawned by suspicious parents

### Network Monitor

Uses eBPF probes on TCP/UDP kernel functions to capture network connections. Tracks:

- Outbound connections (connect), inbound connections (accept), DNS queries, connection close
- Source/destination IP and port, protocol (TCP/UDP/ICMP/RAW), direction, connection state
- Bytes sent and received, connection duration
- DNS query snooping with resolved domain and IP mapping
- GeoIP country lookup
- Private/public IP classification (RFC1918)
- Threat intelligence scoring (0-100)

### File Monitor

Uses eBPF kprobes on VFS functions to capture file system operations within configured watch paths. Tracks:

- File create, write, delete, rename, chmod, and exec operations
- SHA256 hashes before and after modification (when `hash_on_write` is enabled)
- File metadata: size, permissions, inode, device, symlink status, hidden file detection

### Auth Monitor

Monitors authentication-related log sources (e.g., `/var/log/auth.log`, journal) to capture:

- Successful and failed login attempts (SSH, local, GDM, etc.)
- Sudo executions with target user and command
- Authentication method (password, publickey, keyboard-interactive)
- Source IP for remote logins, TTY, service name
- Raw log line for forensic reference

### Command Monitor

Captures interactive shell command execution and shell history:

- Real-time command execution in interactive terminals
- Shell history file changes (.bash_history, .zsh_history, etc.)
- Automatic tagging of suspicious patterns: reverse shells, history evasion, port scanners, credential dumpers, sudo root shells

### Registry Monitor

Monitors critical Linux configuration files for modifications (the Linux equivalent of Windows registry monitoring):

- Watches files like `/etc/passwd`, `/etc/shadow`, `/etc/sudoers`, `/etc/ssh/sshd_config`, `/etc/crontab`, `/etc/ld.so.preload`, and more
- Detects key changes with best-effort config parsing
- Captures old and new values, before/after hashes
- Categorizes changes by type: auth, cron, sudoers, ssh, ld

### Vulnerability Monitor

Periodically inventories installed packages and reports them to the backend:

- Collects package name, version, and architecture from system package managers
- Reports OS name and version
- Backend performs CVE matching against known vulnerability databases

### Browser Monitor

Receives web request events from the TraceGuard browser extension (Chrome/Firefox) via a localhost-only HTTP endpoint.

- Listens on `127.0.0.1:9999` (configurable) for POST requests to `/browser-events`
- Accepts JSON batches from the extension (max 2MB per request)
- Only accepts connections from localhost (127.0.0.1 or ::1)
- Extracts: URL, domain, path, method, status code, content type, referrer, tab URL, resource type, server IP, redirect chain
- Detects form submissions (POST to main_frame) as potential credential submissions
- Auto-tags events: `browser`, resource type, `form-submit`, `cached`, `redirected`, `auth-page`
- Provides a `/health` endpoint for extension connectivity checks

### Kernel Module Monitor (kmod)

Monitors kernel module loads and unloads by polling `/proc/modules`. Detects rootkits and unauthorized kernel drivers.

- Polls `/proc/modules` at configurable intervals (default 5 seconds)
- Detects new module loads and module removals by comparing against a baseline snapshot
- Checks module signing status (signed vs unsigned) via `modinfo`
- Detects kernel taint state changes after module loads
- Reports module name and size

**Event types:** `KERNEL_MODULE_LOAD`, `KERNEL_MODULE_UNLOAD`

**Config options:**

| Key | Type | Default | Description |
|---|---|---|---|
| `monitors.kmod.enabled` | bool | `true` | Enable kernel module monitoring. |
| `monitors.kmod.poll_interval_s` | int | `5` | How often (in seconds) to poll /proc/modules. |

### USB Device Monitor

Monitors USB device connect/disconnect events by polling `/sys/bus/usb/devices/`. Detects mass storage devices (data exfiltration vectors) and rapid USB insertion attacks (BadUSB, rubber ducky).

- Polls sysfs USB device tree at configurable intervals (default 10 seconds)
- Detects new device connections and removals by comparing against a baseline
- Extracts vendor ID, product ID, vendor name, product name, serial number, bus/device numbers
- Classifies device types (mass storage, HID, etc.) from sysfs attributes

**Event types:** `USB_CONNECT`, `USB_DISCONNECT`

**Config options:**

| Key | Type | Default | Description |
|---|---|---|---|
| `monitors.usb.enabled` | bool | `true` | Enable USB device monitoring. |
| `monitors.usb.poll_interval_s` | int | `10` | How often (in seconds) to poll sysfs for USB changes. |

### Memory Injection Monitor (memmon)

Detects suspicious memory operations by polling `/proc/[pid]/maps` for anonymous executable memory regions -- a strong indicator of shellcode injection or reflective DLL loading.

- Scans `/proc/*/maps` at configurable intervals (default 15 seconds)
- Identifies anonymous memory regions with executable permissions (rwxp) not backed by a file
- Maintains a baseline of known regions per PID to avoid duplicate alerts
- Skips processes known to use JIT compilation (Java, Node.js, Python, browsers, VS Code) via configurable ignore list
- Reports region address, size, and permissions

**Event types:** `MEMORY_INJECT`

**Config options:**

| Key | Type | Default | Description |
|---|---|---|---|
| `monitors.memmon.enabled` | bool | `true` | Enable memory injection detection. |
| `monitors.memmon.poll_interval_s` | int | `15` | How often (in seconds) to scan /proc/*/maps. |
| `monitors.memmon.ignore_comms` | string[] | `[java, node, python3, python, firefox, chrome, chromium, code]` | Process names to skip (JIT compilers create legitimate executable anonymous regions). |

### Cron Parsing Monitor (cronmon)

Watches cron directories and systemd timer files for changes. Subscribes to FILE_WRITE and FILE_CREATE events on the event bus, parses crontab content, and emits structured CRON_MODIFY events with entry details and suspicious pattern detection.

- Subscribes to file events on cron paths (no additional polling)
- Parses crontab files to extract schedule, user, and command fields
- Tags suspicious entries: download commands (wget, curl), encoded payloads (base64), reverse shell patterns (/dev/tcp, nc -e, bash -i)
- Monitors standard cron directories: `/etc/crontab`, `/etc/cron.d`, `/etc/cron.daily`, `/etc/cron.hourly`, `/etc/cron.weekly`, `/etc/cron.monthly`, `/var/spool/cron`

**Event types:** `CRON_MODIFY`

**Config options:**

| Key | Type | Default | Description |
|---|---|---|---|
| `monitors.cronmon.enabled` | bool | `true` | Enable cron parsing monitor. |
| `monitors.cronmon.watch_paths` | string[] | `[/etc/crontab, /etc/cron.d, /etc/cron.daily, /etc/cron.hourly, /etc/cron.weekly, /etc/cron.monthly, /var/spool/cron]` | Paths to monitor for cron changes. |

### Named Pipe Monitor (pipemon)

Monitors watched directories for new named pipes (FIFOs). Named pipes in temp directories are a common indicator of C2 frameworks like Cobalt Strike and PsExec for inter-process communication.

- Polls watched directories at configurable intervals (default 10 seconds)
- Detects new FIFO-type files by comparing against a baseline
- Reports pipe path, permissions, and location classification (tmp, dev_shm, run)
- Default watch paths: `/tmp`, `/var/tmp`, `/dev/shm`, `/run`

**Event types:** `PIPE_CREATE`

**Config options:**

| Key | Type | Default | Description |
|---|---|---|---|
| `monitors.pipe.enabled` | bool | `true` | Enable named pipe monitoring. |
| `monitors.pipe.poll_interval_s` | int | `10` | How often (in seconds) to scan directories for new FIFOs. |
| `monitors.pipe.watch_paths` | string[] | `[/tmp, /var/tmp, /dev/shm, /run]` | Directories to watch for named pipe creation. |

### Network Share Monitor (sharemount)

Monitors network filesystem mounts (NFS, CIFS, SMB) by polling `/proc/mounts`. Detects lateral movement via new share mounts and data staging via network shares.

- Polls `/proc/mounts` at configurable intervals (default 10 seconds)
- Detects mount and unmount events for network filesystem types: cifs, nfs, nfs4, smbfs
- Extracts mount source, mount point, filesystem type, mount options, and remote host
- Compares against baseline to detect new mounts and removed mounts

**Event types:** `SHARE_MOUNT`, `SHARE_UNMOUNT`

**Config options:**

| Key | Type | Default | Description |
|---|---|---|---|
| `monitors.share.enabled` | bool | `true` | Enable network share monitoring. |
| `monitors.share.poll_interval_s` | int | `10` | How often (in seconds) to poll /proc/mounts. |

### TLS SNI Monitor (tlssni)

Passively extracts TLS Server Name Indication (SNI) from ClientHello messages using a raw AF_INET socket. This reveals which HTTPS domains every process connects to without certificates or MITM proxy. Requires root or CAP_NET_RAW.

- Opens a raw socket to capture outbound TCP packets to port 443
- Parses TLS ClientHello messages to extract the SNI extension field
- Performs PID attribution by correlating source port with `/proc/net/tcp`
- Deduplicates events with a 30-second TTL cache (same domain + PID)
- Detection rules match on rare TLDs and beaconing patterns

**Event types:** `NET_TLS_SNI`

**Config options:**

| Key | Type | Default | Description |
|---|---|---|---|
| `monitors.tlssni.enabled` | bool | `true` | Enable TLS SNI extraction. |

## Event Bus Architecture

The agent uses a central event bus (`internal/events/bus.go`) that implements a synchronous fan-out pattern with per-subscriber asynchronous buffered channels.

**Key characteristics:**

- **Interface-based:** All monitors publish events implementing the `Event` interface (`EventType() string`, `EventID() string`).
- **Fan-out:** Each published event is delivered to all matching subscribers. Subscribers can register for a specific event type or `"*"` for all events.
- **Buffered channels:** Each subscriber gets a dedicated buffered channel (4096 events deep) and its own consumer goroutine.
- **Non-blocking publish:** If a subscriber's channel is full, the event is dropped and the drop counter is incremented. This prevents a slow consumer from blocking the entire pipeline.
- **Statistics:** The bus tracks total events published, events dropped, and active handler count.
- **Unsubscribe:** `Subscribe()` returns a cleanup function that removes the subscriber and closes its channel.

On startup, the agent subscribes both the local buffer and the gRPC transport to `"*"` (all events), so every event is both persisted locally and streamed to the backend.

## Local SQLite Buffer

The agent maintains a local SQLite database (`/var/lib/edr/events.db` by default) for offline resilience:

- Every event published on the bus is written to SQLite.
- If the gRPC connection to the backend is down, events accumulate in the buffer.
- On reconnection, buffered events are flushed to the backend.
- Maximum buffer size is configurable (`max_size_mb`, default 512 MB). Oldest events are dropped when the limit is reached.
- Flush interval is configurable (`flush_interval_s`, default 5 seconds).

This ensures no events are lost during network outages, backend maintenance, or agent restarts.

## gRPC Transport

The agent streams events to the backend using a client-streaming gRPC connection:

- **Registration:** On connect, the agent calls `Register()` with its ID, hostname, OS, version, and IP.
- **Streaming:** Events are sent via `StreamEvents()`, a client-streaming RPC. The agent sends `EventEnvelope` messages containing JSON-encoded event payloads.
- **Heartbeat:** A unary `Heartbeat()` RPC is called every 30 seconds with agent stats (events sent, events dropped, buffer size, CPU, memory).
- **Reconnection:** On connection loss, the transport reconnects with exponential backoff.
- **TLS:** Supports server-only TLS (verify backend cert) and mutual TLS (client cert). Set `insecure: false` and configure CA/cert/key for production.
- **Live Response:** A background goroutine connects a bidirectional stream for remote command execution from the backend.

## Self-Protection

The agent includes self-protection mechanisms to resist tampering:

- **Watchdog mode** (`self_protect.watchdog: true`): Monitors the agent process and respawns it if killed. Publishes `AGENT_TAMPER` events when tampering is detected.
- **Binary immutability** (`self_protect.immutable_bin: false`): When enabled, applies `chattr +i` on the agent binary to prevent modification or deletion. Requires root.
- **Auto-detection:** If `bin_path` is empty, the agent auto-detects its own binary path from `/proc/self/exe`.

## Severity Levels

Events carry a severity level (0-4):
